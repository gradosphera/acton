---
title: "LiteAPI Support"
description: "LiteServer binary API coverage in Acton local development node"
---

Acton localnet exposes a `LiteServer`-compatible binary endpoint for tools that
talk to TON nodes through LiteAPI instead of TonCenter HTTP APIs. The endpoint is
intended for local indexing, ABI discovery, and get-method execution against the
localnet state. It is not a full validator liteserver yet.

## Endpoint

When `acton localnet start` runs on HTTP port `5411`, LiteAPI listens on TCP port
`5412`. In general, the LiteAPI port is always the localnet HTTP port plus one.

Startup output prints both values:

```text
Starting Localnet server and UI on http://127.0.0.1:5411
Starting Localnet LiteAPI on tcp://127.0.0.1:5412
    Key LiteAPI public key: <base64-ed25519-public-key>
```

Use the printed TCP address and public key in client configs. For TON global
config JSON, `127.0.0.1` is usually encoded as integer `2130706433`.

```json
{
  "liteservers": [
    {
      "ip": 2130706433,
      "port": 5412,
      "id": {
        "@type": "pub.ed25519",
        "key": "<base64-ed25519-public-key>"
      }
    }
  ]
}
```

<Callout type="info">
The localnet LiteAPI key is stable across runs, but the docs intentionally show a
placeholder. Copy the key from the current `acton localnet start` output.
</Callout>

## Advertised Capabilities

`liteServer.getVersion` and `liteServer.getMasterchainInfoExt` currently report:

| Field | Value | Meaning |
|-------|-------|---------|
| `version` | `0x101` | LiteServer protocol version `1.1`. |
| `capabilities` | `6` | `+2` for `masterchainInfoExt`, `+4` for `runSmcMethod`. |

Localnet does not advertise capability `+1` because block proof-chain
construction is not implemented yet. Clients that require secure proof-chain
validation must either use unsafe/no-proof mode for local development or wait
for proof-chain support.

## Supported Methods

| Method | Status | Notes |
|--------|--------|-------|
| `liteServer.getMasterchainInfo` | Supported | Returns the latest localnet block id and zero-state id. |
| `liteServer.getMasterchainInfoExt` | Supported | Reports localnet version, capabilities, latest block, and time metadata. |
| `liteServer.getTime` | Supported | Returns current wall-clock Unix time. |
| `liteServer.getVersion` | Supported | Reports version `0x101` and capabilities `6`. |
| `liteServer.getBlock` | Supported | Returns the localnet block BoC for the requested seqno. |
| `liteServer.getBlockHeader` | Limited | Returns block id and requested header flags; `header_proof` is empty. |
| `liteServer.sendMessage` | Supported | Accepts an external-in message BoC and queues it for localnet execution. |
| `liteServer.getAccountState` | Limited | Returns account state data for the requested block; proof payloads are synthetic/minimal. |
| `liteServer.getAccountStatePrunned` | Limited | Same localnet behavior as `getAccountState`. |
| `liteServer.runSmcMethod` | Limited | Supports mode `0` and result bit `4`; proof, `init_c7`, and library-extra modes are rejected. |
| `liteServer.getShardInfo` | Limited | Returns the local single-shard descriptor; proof payload is empty. |
| `liteServer.getAllShardsInfo` | Limited | Returns local all-shards data; proof payload is empty. |
| `liteServer.getOneTransaction` | Limited | Returns transaction data by account and logical time; proof payload is empty. |
| `liteServer.getTransactions` | Supported | Returns account transactions as a multi-root BoC plus containing block ids. |
| `liteServer.lookupBlock` | Supported | Resolves by seqno, logical time, or Unix time in the single localnet shard. |
| `liteServer.listBlockTransactions` | Limited | Supports `count` and `after`; reverse order and proof mode are not implemented. |
| `liteServer.listBlockTransactionsExt` | Limited | Supports `count` and `after`; returns transaction cells as a multi-root BoC. |
| `liteServer.getConfigAll` | Limited | Returns local config data with a synthetic/minimal proof payload. |
| `liteServer.getConfigParams` | Limited | Returns local config data; `param_list` filtering is not implemented yet. |
| `liteServer.getLibraries` | Supported | Returns locally known public libraries for requested hashes. |
| `liteServer.getLibrariesWithProof` | Limited | Returns library data; proof payloads are empty. |
| `liteServer.getShardBlockProof` | Limited | Returns a minimal shard-block link with an empty proof payload. |
| `liteServer.waitMasterchainSeqno` wrapper | Supported | Waits until localnet reaches the requested seqno or the request timeout expires. |

`runSmcMethod` uses the same execution path as localnet `runGetMethod`: it reads
code, data, balance, libraries, previous block info, and virtual time from the
requested localnet block. Missing or non-runnable accounts are reported with
LiteServer-style exit code `-256`.

## Unsupported Methods

Unsupported methods return `liteServer.error`.

| Method | Status | Reason |
|--------|--------|--------|
| `liteServer.getState` | Not supported | Full block-state BoC export is not implemented. |
| `liteServer.lookupBlockWithProof` | Not supported | Depends on proof-chain support. |
| `liteServer.getBlockProof` | Not supported | Depends on real block proof-chain construction. |
| `liteServer.getValidatorStats` | Not supported | Localnet does not emulate validator statistics. |
| `liteServer.getOutMsgQueueSizes` | Not supported | Out-message queue size snapshots are not modeled. |
| `liteServer.getBlockOutMsgQueueSize` | Not supported | Block queue proof/data is not modeled. |
| `liteServer.getDispatchQueueInfo` | Not supported | Dispatch queue state is not modeled. |
| `liteServer.getDispatchQueueMessages` | Not supported | Dispatch queue messages are not modeled. |
| `liteServer.nonfinal.getValidatorGroups` | Not supported | Non-final validator group API is outside the current localnet model. |
| `liteServer.nonfinal.getCandidate` | Not supported | Non-final candidate API is outside the current localnet model. |
| `liteServer.nonfinal.getPendingShardBlocks` | Not supported | Pending shard-block candidate API is outside the current localnet model. |

## Current Limitations

The current implementation is designed for local indexing and execution tooling,
not trustless validator-grade verification.

- Localnet has one real executable shard block stream in workchain `0`. LiteAPI
  exposes deterministic synthetic masterchain ids in workchain `-1` only as a
  compatibility anchor for clients that discover shard blocks through
  masterchain calls.
- Synthetic masterchain ids are not backed by independent masterchain block BoCs.
  Real transaction data lives on the workchain `0` shard block for the same
  seqno; synthetic masterchain transaction lists are empty.
- Proof fields are minimal parsing placeholders, or omitted where the client TL
  type requires omission. They do not prove membership from a trusted masterchain
  root and must not be used with secure proof checking enabled.
- Generated block BoCs are real TON `Block` cells with transaction data and a
  `state_update`, but they intentionally omit validator consensus artifacts:
  signatures, key-block proof chains, shard split/merge history, validator
  statistics, dispatch queues, and out-message queue proofs.
- `runSmcMethod` supports only the plain execution/result response path. Proof,
  `init_c7`, and library-extra response modes are rejected.
- Block transaction listing supports `count` and `after`; reverse order,
  metadata proof expansion, and proof mode are not implemented yet.
- `getConfigParams` currently returns the local config payload without applying
  `param_list` filtering.
- Full `liteServer.getState` export is not available yet, even though generated
  blocks contain state hashes in `state_update`.

## Proof Policy

Many clients can run LiteAPI calls with disabled proof validation for local
development. That mode is currently required for localnet data that depends on
proof fields, because localnet does not yet build validator-grade Merkle proofs
or block proof chains.

Use secure proof-checking only after the needed proof methods are listed as
supported without the "Limited" status on this page.
